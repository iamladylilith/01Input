VS Code Learning Plan

Your roadmap from first launch to writing Python that actually does your job. Built for data people, not developers.

6
Steps
6w
Timeline
0
Experience needed
01
Get set up
Install VS Code and open your first file Day 1
What to do
  • Download from code.visualstudio.com and install
  • File → Open Folder — always work in a folder
  • Click + in the sidebar to create a file, name it hello.py
  • Save constantly with Ctrl+S
Why it matters
VS Code treats folders as projects — it keeps files organised and lets Python find them properly. Think of it like always working in a labelled binder, not loose sheets on a desk.
02
Learn 7 shortcuts — ignore the rest
These are the only ones you need daily Day 2–3
What to do
  • Ctrl+S — save the file (use obsessively)
  • Ctrl+Z — undo
  • Ctrl+/ — comment a line
  • Ctrl+P — open any file by name
  • Ctrl+Shift+P — command palette
  • Alt+↑/↓ — move line up or down
  • Ctrl+` — open terminal
Why it matters
The command palette (Ctrl+Shift+P) is the secret weapon. If you ever wonder “can VS Code do X?” type it there. It finds settings, extensions, and actions all in one place. Experienced developers use it constantly.
03
Install 3 extensions — no more
Everything you need for data work Day 3
What to do
  • Python by Microsoft — makes VS Code understand Python
  • Pylance — smarter autocomplete as you type
  • Rainbow CSV — colours spreadsheet columns so data files are readable
Why it matters
You will see hundreds of extensions. Ignore them. More extensions means more things to configure and break. These three are everything you need for the next two months. Add more only when you hit a specific problem they would solve.
04
Write your first real script
Not hello world — something actually useful Week 1–2
What to do
  • Open a CSV and count how many rows it has
  • Flag any rows where a required field is empty
  • Count unique values in a column
  • Run it: open terminal, type python hello.py and press Enter
Why it matters
The gap between tutorial Python and Python that does your job is smaller than you think. Use a real file from work — even a tiny one. The motivation difference when code actually helps you is enormous.
05
Learn to read error messages
Errors are instructions, not failures Week 2–3
What to do
  • Read the last line of any error first — that is the real problem
  • SyntaxError means typo or missing colon
  • NameError means variable name does not exist yet
  • FileNotFoundError means wrong file path
  • Copy any error into Google — it has been solved before
Why it matters
Every beginner panics at red text. Every experienced developer reads it calmly. The only difference is familiarity. The day you stop feeling anxious about errors is the day you actually become a programmer.
06
Build something for your actual job
This is when everything clicks Week 4–6
What to do
  • Check a D365 export for missing or malformed vendor fields
  • Compare two versions of a master data file and highlight what changed
  • Merge multiple CSVs from a folder into one clean file
  • Generate an automated summary report
Why it matters
When code saves you 2 hours at work two things happen — you know this is worth continuing, and you have a story to tell in your next interview. Both matter more than you would think.
